### Task D4 — 数据库定价

#### 要讲清的问题

> demo 库里有历史票价（`segments.price`），但**没有统一的定价规则表**。D4 要新建 `bookings.pricing_rules`，供统计和验证；D6 订票则按「飞行分钟数 × 舱位单价」算价。

#### 两个文件

| 文件 | 作用 |
|------|------|
| `price.sql` | 建表 + 从历史数据填充 |
| `d4.sql` | 对比计算价与实际价 |

#### 三者关系（答辩常问）

| 层次 | 谁创建 | 内容 |
|------|--------|------|
| 库 `demo` | Docker / 本机 | 空壳库名 |
| `bookings.*` 航班表 | 导入 PostgresPro demo | `airports_data`、`flights`、`segments`… |
| `pricing_rules` | **执行 `price.sql`** | D4 产出，**不在原始 demo 里** |

#### 现场演示

```powershell
docker exec -it postgres_db_demo psql -U postgres -d demo -c "\dt bookings.*"
docker exec -it postgres_db_demo psql -U postgres -d demo -c "SELECT COUNT(*) FROM bookings.pricing_rules;"
```

讲词：`\dt` 能看到 `pricing_rules` 说明 D4 已执行；行数应大于 0。

#### 指代码 — `Task D4/price.sql`

指着 **INSERT … SELECT** 的三表 JOIN：

```sql
FROM bookings.routes r
JOIN bookings.flights f ON r.route_no = f.route_no
    AND r.validity @> f.scheduled_departure   -- 关键：时效关联
JOIN bookings.segments s ON f.flight_id = s.flight_id
WHERE f.status = 'Arrived'
```

> 新版库同一条 `route_no` 有不同时期版本，必须加 `validity @>`，否则 join 到错误航线。  
> 单价：Economy 50 / Comfort 65 / Business 100，与 D6 `bookings.js` 的 `RATE` 一致。

#### 指索引 — `postgres-init/indexes.sql`

> 给 `scheduled_departure`、起降机场、登机牌 `ticket_no` 等常查列建索引，加速 `/routes` 和值机。

#### 演示 ② 航线搜索（核心）

**URL（已验证）：**

```
http://localhost:3000/routes?from=Moscow&to=LED&date=2025-12-31&connections=1&bookingClass=economy
```

**指 `routes/routes.js`：**

1. **参数**：`isAirportCode` 区分三字码与城市名；`connections=unbound` 最多 10 次中转  
2. **锚点**（递归第一段）：当天从起点出发的第一程，`connections=0`  
3. **递归**（`UNION ALL`）：落地机场接下一程；转机 ≥40 分钟、≤24 小时；`path_airports` 防绕圈  
4. **外层过滤**：`WHERE arrival_city = ...`（注意：这里不能写 `i.`，外层没有别名 `i`）  
5. **二次查询**：把 `flight_id` 数组拼成 `segments` JSON

## 四、Q&A

### Q1：`validity @>` 是什么？

> 航线有时效版本，航班起飞时间必须落在 `validity` 区间内，才能 join 到正确那一版航线。

### Q2：递归 CTE 干什么用？

> 中转搜索：先找第一程，再在落地机场递归找下一程，直到到达终点或超过 `connections` 上限。

### Q3：为什么用事务订票？

> `bookings`、`tickets`、`segments` 三表必须同时成功或同时回滚，避免脏数据。

### Q4：值机为什么只传 `ticket_no`？

> 票号足以从 `segments` 反查航班；自动分配座位；重复请求返回已有登机牌（幂等）。

### Q5：为什么扁平化而不是 MVC？

> 8 个接口规模下，扁平 `routes/` 更易读易答辩；规模大后再拆层。

### Q6：`\l` 有 demo 但没有 `pricing_rules`？

> `demo` 只是库名；航班表靠导入 PostgresPro；`pricing_rules` 靠执行 `price.sql`。

### Q7：`missing FROM-clause entry` 是什么？

> SQL 里引用了尚未 JOIN 的表别名（开发中 `routes.js` 曾有两处：`arr_a2` 顺序、`i.` 外层别名），与 demo 是否导入无关。修复后 `/routes` 正常返回 JSON。

### Q8：索引怎么选？

> 看 WHERE 常用列：`scheduled_departure`、起降机场、`ticket_no` 等。
