# 航空客运预订系统 — 答辩报告

## 一、项目背景

本项目基于 PostgresPro 提供的航空运输演示数据库（**2025-09-01 版本**，PostgreSQL ≥ 15）。该版本相较旧版有大量 breaking change，正确适配这些变更是本次开发的核心挑战之一。

关键 Schema 变更对照：

| 旧版 | 2025 新版 | 影响 |
|------|-----------|------|
| `ticket_flights` | `segments` | 所有航段查询、插入 |
| `aircrafts` | `airplanes_data` | 机型信息查询 |
| `airports`（视图） | `airports_data`（表，jsonb） | 城市/机场名需 `->>'en'` 提取 |
| `amount` | `price` | 航段价格字段 |
| `aircraft_code` on `flights` | `airplane_code` on `routes` | 获取机型必须做 temporal join |
| 无时间维度 | `routes.validity @> flights.scheduled_departure` | 所有 routes join 必须加时间条件 |
| 无 `outbound` | `tickets.outbound boolean` | 区分去程/返程票 |

## 二、Task D4 — 数据修复与定价规则

### 目标

历史数据库中只有每张票的实际售价记录（`segments.price`），没有规范化的定价表。Task D4 的目标是从历史数据中还原出每条航线（`route_no`）× 每种舱位（`fare_conditions`）的基准价格，写入 `bookings.pricing_rules`，供未来订票使用。

### 实现思路

脚本分 6 步，全部用纯 SQL 的临时表流水线完成：

**Step 1 — 提取历史均价**

```sql
SELECT f.route_no, s.fare_conditions, bp.seat_no, AVG(s.price) AS price
FROM bookings.segments AS s
JOIN bookings.boarding_passes AS bp ON ...
JOIN bookings.flights AS f ON ...
GROUP BY f.route_no, s.fare_conditions, bp.seat_no
```

从 `segments`（航段价格）+ `boarding_passes`（座位号）+ `flights`（航线号）三表关联，得到每个 `(route_no, fare_conditions, seat_no)` 的历史均价。

**Step 2 — 枚举所有理论座位**

```sql
SELECT DISTINCT r.route_no, se.seat_no, se.fare_conditions
FROM bookings.routes AS r
JOIN bookings.seats AS se ON se.airplane_code = r.airplane_code
```

注意：这里**不需要 JOIN `airplanes_data`**，因为 `airplane_code` 已经在 `routes` 表上，直接用即可。这也是排查出的一个 bug——旧版实现错误地 JOIN 了不存在的 `airplanes` 表。

**Step 3 — LEFT JOIN 得到含空缺的价格表**

将步骤1和步骤2 LEFT JOIN，没有历史销售记录的座位价格标为 0，同时提取座位行号（`row_no`）用于后续插值。

```sql
CASE WHEN LENGTH(seat_no) = 3 THEN SUBSTRING(seat_no FROM 1 FOR 2)
     ELSE SUBSTRING(seat_no FROM 1 FOR 1) END AS row_no
```

**Step 4 — 同排插值**

对价格为 0 的座位，在同一航线、同一舱位、同一排中找有价格的座位，取最大值填充：

```sql
MAX(pn.price) FROM partial_price pz JOIN partial_price pn
ON pn.route_no = pz.route_no AND pn.fare_conditions = pz.fare_conditions
   AND pn.row_no = pz.row_no AND pn.price > 0
WHERE pz.price = 0
```

**Step 5 — 合并完整价格**

用 `COALESCE` 将插值结果合并回原始价格表，得到每个座位的完整价格。

**Step 6 — 聚合写入 pricing_rules**

```sql
SELECT route_no, fare_conditions, AVG(price) AS base_price, COUNT(seat_no) AS seat_count
FROM full_price WHERE price > 0
GROUP BY route_no, fare_conditions
```

最终结果：**1410 行**，覆盖所有航线的所有舱位组合，加 PRIMARY KEY 和索引后清理所有中间表。

### 幂等性设计

脚本头部加了 `DROP TABLE IF EXISTS`，可以安全地重复执行：

```sql
DROP TABLE IF EXISTS routes_seats_price, routes_with_seats, partial_price, restored_price, full_price;
DROP TABLE IF EXISTS bookings.pricing_rules;
```

## 三、Task D5 — API 设计（OpenAPI 3.0）

### 设计原则

- 资源导向的 URL 设计，名词复数
- 查询参数做分页（`limit` / `page`），有合理的最大值限制
- 所有错误统一 `{ "message": "..." }` 格式
- `fareConditions` 用枚举约束：`Economy` / `Comfort` / `Business`

### 接口清单

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/cities` | 分页获取所有城市（去重，按字母排序） |
| GET | `/cities/{cityName}/airports` | 获取指定城市的机场列表 |
| GET | `/airports` | 分页获取所有机场 |
| GET | `/airports/{airportCode}/schedules` | 机场进港/出港时刻表（`type=inbound\|outbound`） |
| GET | `/flights` | 搜索两点间航班（支持中转、舱位筛选） |
| POST | `/bookings` | 为单名乘客创建预订 |
| POST | `/checkins` | 办理值机（自动分配座位，生成登机牌） |

### 关键设计决策

**`/flights` 的 `transfers` 参数**：取值 `0`（直飞）、`1`、`2`、`3`，不传表示不限（后端默认最多 3 次），最大值限制为 3，防止递归 CTE 深度过大影响性能。

**`/checkins` 只需 `ticketNo`**：票号唯一标识一个 `(乘客, 航班)` 组合，后端直接从 `segments` 查出 `flight_id`，无需客户端传入内部 ID。对外暴露数据库内部 `flight_id` 是不良实践。

**`/bookings` 的字段**：包含 `flightDate`（航班日期，用于查询航班）、`flightNo`（航线号）、`fareConditions`、`passengerId`、`passengerName`、`outbound`（可选，默认 `true`，支持创建返程票）。

### 核心实现：航班搜索（Flight.getFlights）

航班搜索分两个阶段：

**1. 解析出发/到达点**

`origin` 和 `destination` 可以是城市名或机场代码，统一解析为机场代码数组：

```sql
SELECT airport_code FROM bookings.airports_data
WHERE airport_code = $1 OR city->>'en' = $1;
```

**2. 直飞查询**

标准 JOIN，关键是 temporal join 获取机型信息：

```sql
FROM bookings.flights AS f
JOIN bookings.routes AS r
  ON r.route_no = f.route_no AND r.validity @> f.scheduled_departure
JOIN bookings.airplanes_data AS ap ON ap.airplane_code = r.airplane_code
```

`scheduled_duration` 直接用 `scheduled_arrival - scheduled_departure` 计算，不从 `routes` 取（避免时区转换问题）。

**3. 中转航班 — 递归 CTE**

这是技术难度最高的部分。用 PostgreSQL 的 `WITH RECURSIVE` 在航班图上做深度优先遍历：

```sql
WITH RECURSIVE flight_paths AS (
  -- 基础：从出发机场当天出发的所有航班
  SELECT ..., ARRAY[f.flight_id] AS path_ids,
              ARRAY[r.departure_airport::text] AS path_airports,
              1 AS depth
  FROM flights JOIN routes ...
  WHERE r.departure_airport = ANY($1)
    AND f.scheduled_departure::date = $3::date

  UNION ALL

  -- 递归：从上一程落地点继续延伸
  SELECT ..., fp.path_ids || f2.flight_id,
              fp.path_airports || r2.departure_airport::text,
              fp.depth + 1
  FROM flight_paths fp
  JOIN routes r2 ON r2.departure_airport = fp.arrival_airport
  JOIN flights f2 ON ...
    AND f2.scheduled_departure >= fp.scheduled_arrival + INTERVAL '30 minutes'  -- 最短中转时间
    AND f2.scheduled_departure < fp.scheduled_departure + INTERVAL '2 days'    -- 最长等待时间
    AND NOT (f2.flight_id = ANY(fp.path_ids))        -- 避免重复航班
    AND NOT (r2.arrival_airport = ANY(fp.path_airports))  -- 避免重复经停城市
  WHERE fp.depth < $N   -- 深度上限 = transfers 参数
    AND NOT (fp.arrival_airport = ANY($2))  -- 剪枝：已到终点则停止
)
```

两个数组 `path_ids` 和 `path_airports` 分别用于去重检测，防止出现环路。`INTERVAL '30 minutes'` 保证最短中转时间可行。

最终结果：直飞 + 中转合并返回，按出发时间排序。

### 核心实现：创建预订（Booking.createBooking）

全程在一个数据库事务中完成，保证原子性：

```
BEGIN
  1. 查航班（temporal join 获取 airplaneCode + routeNo）
  2. 检查航班状态（Arrived/Cancelled → 400）
  3. 计算剩余座位数：总座位数 - 已订座位数
  4. 查定价（优先 pricing_rules，SAVEPOINT 保护，降级到 segments 历史数据）
  5. 生成 bookRef（UUID v4 → 取前 6 位大写）
  6. INSERT bookings
  7. 生成 ticketNo（UUID v4 → 取前 13 位）
  8. INSERT tickets（含 outbound 字段）
  9. INSERT segments（新表名，price 新字段名）
COMMIT / ROLLBACK
```

**定价降级机制**（SAVEPOINT 设计）：

```javascript
try {
  await transaction.query('SAVEPOINT price_lookup');
  // 查 pricing_rules（D4 的输出）
  result = await transaction.query('SELECT base_price FROM bookings.pricing_rules ...');
  await transaction.query('RELEASE SAVEPOINT price_lookup');
} catch (_) {
  // 表不存在时回滚到 SAVEPOINT，不影响外层事务
  await transaction.query('ROLLBACK TO SAVEPOINT price_lookup');
  // 降级：从 segments 历史数据取最低价
  result = await transaction.query('SELECT price FROM bookings.segments ...');
}
```

这保证了即使 D4 脚本没有执行，预订接口也能正常工作。

### 核心实现：值机（Flight.checkIn）

值机流程同样在事务中，步骤顺序经过仔细设计：

```
BEGIN
  1. 用 ticketNo 查 segments → 得到 flightId 和 fareConditions
     （票不存在 → 404 NotFound）
  2. 用 flightId 查 flights + routes → 验证航班状态 In('On Time','Delayed','Boarding')
     （状态不对 → 404 NotFound）
  3. 检查是否已值机过
     （已值机 → 400 BadRequest）
  4. 从 seats 中排除 boarding_passes 已占用的，取字母序第一个可用座位
     （无座位 → 400 BadRequest）
  5. 生成 boardingNo = MAX(boarding_no) + 1
  6. INSERT boarding_passes
COMMIT
```

**关键设计**：只传 `ticketNo`，不传 `flightId`。原因：`segments.(ticket_no, flight_id)` 是联合主键，一张票只对应一个航班，后端自己查即可。对外暴露数据库内部整数 ID 是不良实践。

### 入参校验（Celebrate + Joi）

每个接口在路由层前置校验，校验失败直接被 `errorHandler` 捕获为 400：

| 参数 | 校验规则 |
|------|---------|
| `ticketNo` | `string().length(13)` |
| `flightNo` | `string().length(6)` |
| `departureDate` | `pattern(/^\d{4}-\d{2}-\d{2}$/)` |
| `fareConditions` | `valid('Economy','Comfort','Business')` |
| `transfers` | `integer().min(0).max(3)` |
| `limit` | `number().min(1).max(100).default(10)` |

### 错误处理

三层错误链：
1. Celebrate 校验失败 → `isCelebrateError` 判断 → 包装为 `BadRequest`
2. 已知业务错误（`http-errors` 实例）→ 直接取 `error.status` + `error.message`
3. 未知错误 → 包装为 `InternalServerError`，统一格式 `{ message }`

### 推荐索引

```sql

-- 航段查询

-- airports_data 城市名（jsonb 表达式索引）
```

## 五、排查与修复的问题

答辩时重点可以提这几个，体现了对 Schema 的深度理解：

**1. D4 的多余 JOIN**
原版 `routes_with_seats` 写了 `JOIN bookings.airplanes`，但 `airplane_code` 已经在 `routes` 上，完全不需要这个 JOIN，且 `airplanes` 是视图而非表，在某些环境下可能不存在。移除后逻辑等价，且更健壮。

**2. 值机接口暴露内部 ID**
原版接口要求传 `flightId`（数据库内部整数），D5 设计的是 `flightNo`（航线号字符串）。但实际上 `ticketNo` 已经唯一确定了 `(乘客, 航班)` 的关联，后端从 `segments` 查一次就能得到 `flightId`，无需客户端传任何 flight 标识，接口更干净。

**3. Temporal Join 的必要性**
`flights` 表上没有 `airplane_code`，它在 `routes` 表上。但 `routes` 是有时效的（`validity tstzrange`），同一条航线在不同时期可能使用不同机型。因此查机型必须用：
```sql
JOIN routes r ON r.route_no = f.route_no AND r.validity @> f.scheduled_departure
```
如果只用 `route_no` 做等值 JOIN 会返回多行，导致查询结果错误。

**4. SAVEPOINT 保护定价降级**
在事务中如果直接 `try/catch` 一个失败的查询，PostgreSQL 会把整个事务标记为 aborted 状态，后续所有语句都会报 `ERROR: current transaction is aborted`。用 `SAVEPOINT` 可以在不终止外层事务的前提下回滚单条语句。

## 六、答辩常见问题参考

**Q: 为什么不用 ORM（如 Sequelize/Prisma）？**
A: 本项目的核心查询（递归 CTE、temporal join、jsonb 提取）都是 PostgreSQL 特有语法，ORM 无法直接表达，强行用 ORM 反而会增加复杂度。直接写 SQL 更透明、性能更可控。

**Q: 递归 CTE 性能如何？**
A: 最大深度限制为 3（`transfers ≤ 3`），配合 `idx_flights_route_departure` 索引，每次递归展开只扫描特定日期的特定航线，查询集合可控。对于小型数据集（3个月版本）实测响应在 1 秒以内。

**Q: 如何保证并发预订不超卖？**
A: 预订逻辑在数据库事务中执行，查询剩余座位数和插入 segment 是同一个事务。PostgreSQL 的默认隔离级别（Read Committed）加上行级锁，能防止超卖。如需更严格的隔离，可升级到 Serializable 级别。

**Q: pricing_rules 表里有些航线+舱位组合没有数据怎么办？**
A: `getTicketPrice` 有两级降级：先查 `pricing_rules`，没有再从该航班已有的 `segments` 历史数据取最低价。两级都没有则返回 400（提示无定价信息）。

**Q: `bookings.now()` 是什么？**
A: 演示数据库内置函数，返回数据的"当前时间"快照（2025-12-01 00:00:00+03），以此区分"历史航班"和"未来航班"。本项目不依赖这个函数，而是直接用传入的日期参数查询。
