## 一、开场（约 1 分钟）

> 本项目是一个**航班查询与预订后端**：查城市/机场/航线、订票、在线值机。  
> 基于 PostgresPro 航空 demo 库，分三步完成：**D4 补定价规则 → D5 用 OpenAPI 定契约 → D6 用 Node.js 实现**。  
> 今天重点讲 **D5 接口设计**（尤其是新增筛选功能）和 **D6 整体实现**。

D4 一句话：demo 有历史票价但没有统一规则表，D4 新建 `bookings.pricing_rules`，单价 50/65/100 来自历史数据反推（详见文末 Q9）。

## 二、Task D5 — `api.yaml` 详解（重点）

### 2.1 为什么先写契约再写代码

`Task D5/api.yaml` 是 **OpenAPI 3.0.3** 规范文件，作用：

1. **前后端对齐** — 路径、参数、返回 JSON 形状在编码前定死
2. **可机器读取** — Swagger Editor 可视化；可自动生成客户端/服务端骨架
3. **D6 的实现依据** — `Task D6/api.yaml` 与之同步，每个 `paths` 条目对应一个 `routes/*.js`

**演示方式：** 浏览器打开 Swagger Editor，粘贴 `Task D5/api.yaml`，左侧看结构，右侧看交互文档。

---

### 2.2 文档整体结构

```yaml
openapi: 3.0.3          # 规范版本（见下方「新增①」）
info:                   # 标题、描述、版本号
paths:                  # 8 个 HTTP 接口
components:             # 可复用片段（见下方「新增②」）
  parameters:           #   公共查询/路径参数
  responses:            #   公共响应（如 400）
  schemas:              #   公共数据模型
```

相比早期草稿，当前版本做了这些**结构性改进**：

| 改进 | 说明 |
|------|------|
| **新增① OpenAPI 3.0.3** | 使用 `openapi: 3.0.3`；HTTP 状态码写成 `"200"`、`"400"` 字符串（Swagger Editor 不再报错） |
| **新增② `components` 复用** | 筛选参数、错误响应、数据模型抽到 `components`，各 path 用 `$ref` 引用，避免重复 |
| **新增③ 统一错误格式** | 所有 400 响应引用 `#/components/responses/BadRequest`，body 为 `{ "error": "..." }` |
| **新增④ 完整 Schema** | 每个接口的 200 响应都定义了 `schema` + `example`，而不只是文字描述 |

### 2.4 新增功能重点 — 出港/进港筛选（老师要求的核心改动）

**问题：** 原版 `/airports/{code}/outbound` 一次返回该机场全部出港航线，数据量大、前端难用。

**改动：** 为 **inbound** 和 **outbound** 增加可选查询参数，在契约层写死规则，D6 按契约实现。

#### outbound 新增参数

| 参数 | 类型 | 必填 | 说明 | 示例 |
|------|------|------|------|------|
| `destination` | string | 否 | 目的地：IATA 三字码 **或** 英文城市名 | `LED` 或 `Saint Petersburg` |
| `time_from` | string | 否 | 时刻下限，`HH:MM:SS` | `06:00:00` |
| `time_to` | string | 否 | 时刻上限，`HH:MM:SS` | `18:00:00` |
| `limit` | integer | 否 | 返回条数，默认 10，最大 100 | `5` |

#### inbound 新增参数

| 参数 | 类型 | 必填 | 说明 | 示例 |
|------|------|------|------|------|
| `origin` | string | 否 | 出发地：机场码或城市名 | `Moscow` |
| `time_from` | string | 否 | 同 outbound | `06:00:00` |
| `time_to` | string | 否 | 同 outbound | `23:59:59` |
| `limit` | integer | 否 | 同 outbound | `10` |

#### 契约里怎么写的（指 `api.yaml`）

**① 公共参数定义** — 在 `components/parameters` 里统一定义，path 里 `$ref` 引用：

```yaml
# outbound 用 DestinationFilter
DestinationFilter:
  name: destination
  in: query
  description: 按目的地过滤（机场码或城市名）

# inbound 用 OriginFilter
OriginFilter:
  name: origin
  in: query
  description: 按出发地过滤

# 时间、条数同理
TimeFrom:   { name: time_from, pattern: "^([01][0-9]|2[0-3]):..." }
TimeTo:     { name: time_to,   ... }
ResultLimit: { name: limit, minimum: 1, maximum: 100, default: 10 }
```

**② path 引用参数** — 以 outbound 为例：

```yaml
/airports/{code}/outbound:
  get:
    parameters:
      - $ref: "#/components/parameters/AirportCode"
      - $ref: "#/components/parameters/DestinationFilter"
      - $ref: "#/components/parameters/TimeFrom"
      - $ref: "#/components/parameters/TimeTo"
      - $ref: "#/components/parameters/ResultLimit"
```

**③ 返回 Schema** — 筛选后仍返回 `OutboundRoute` / `InboundRoute` 数组，字段包括 `route_no`、`days_of_week`、`scheduled_time`、目的地/出发地机场与城市。

**④ 400 校验** — `time_from`/`time_to` 格式不对时返回 `BadRequest`（D6 在 SQL 拼装前校验）。

**讲词示例：**

> 筛选不是 D6 临时加的，而是先在 D5 契约里定义好参数名、类型、默认值和上限，D6 的 `airports.js` 只是忠实实现。这样前端看 Swagger 就知道能传什么，后端也有据可查。

**演示 URL：**

```
http://localhost:3000/airports/SVO/outbound?destination=LED&time_from=06:00:00&time_to=18:00:00&limit=5
http://localhost:3000/airports/SVO/inbound?origin=Moscow&limit=10
```

### 2.5 其余接口契约要点

#### Location — 基础查询

| 接口 | 返回形状 | 要点 |
|------|----------|------|
| `GET /cities` | `string[]` | 城市英文名列表 |
| `GET /cities/{city}/airports` | `AirportBrief[]` | `airport_code` + `airport_name` |
| `GET /airports` | `AirportFull[]` | 加 `city` 字段 |

#### Search — 航线搜索

```
GET /routes?from=Moscow&to=LED&date=2025-12-31&connections=1&bookingClass=economy
```

| 参数 | 必填 | 说明 |
|------|------|------|
| `from` / `to` | 是 | 城市名或 IATA 三字码 |
| `date` | 是 | `YYYY-MM-DD` |
| `connections` | 否 | 最大中转次数，或 `unbound` |
| `bookingClass` | 否 | `economy` / `comfort` / `business` |

返回 `RouteOption[]`：每项含 `connectionsCount` 和 `segments` 数组。

#### Booking — 订票

`POST /bookings`，body 为 `BookingRequest`：

- `passenger_id`、`passenger_name`
- `fare_condition`：`Economy` / `Comfort` / `Business`
- `outbound`：boolean，去程/返程标记
- **`segments` 数组** — 支持联程，每段含 `routeNo`、起降机场、出发时间

成功 **201**，返回 `{ book_ref, ticket_no }`。

#### Check-in — 值机

`POST /check-in`，body 只需 `{ "ticket_no": "..." }`。

- 首次值机 → **201** + 登机牌数组
- 已值机重复请求 → **200** + 已有登机牌（幂等）

## 三、Task D6 — 整体讲解（重点）

### 3.1 目录结构与设计理念

```
Task D6/
├── api.yaml                 # 与 D5 同步
├── docker-compose.local.yaml
├── SETUP.md
├── requests.txt
└── app/
    ├── main.js              # 入口
    ├── db.js                # 连接池
    ├── asyncHandler.js      # 错误包装
    ├── Dockerfile
    └── routes/              # ← 核心业务（答辩指读区）
        ├── index.js
        ├── cities.js
        ├── airports.js
        ├── routes.js
        ├── bookings.js
        └── checkin.js
```

**架构选择：扁平 `routes/`，不用 MVC 分层。**

| 原模板结构 | 当前结构 | 原因 |
|------------|----------|------|
| `app.js` + `server.js` | 单一 `main.js` | 8 个接口，一个入口足够 |
| `controllers/` + `models/` | `routes/` 内联 SQL | 读代码时「接口 → SQL」一条线，答辩更清晰 |
| 多层 middleware | `asyncHandler` + 全局 error handler | 够用且少文件 |

依赖只有 **`express` + `pg`**，无 ORM、无多余框架。

---

### 3.2 基础设施层

#### `main.js` — 唯一入口

职责：

1. 创建 Express 实例，`express.json()` 解析 POST body
2. **按顺序挂载** 6 个路由模块
3. **全局错误处理** — 返回 JSON `{ error, hint? }`；识别 demo 未导入（`42P01`）并给出 SETUP 提示
4. **启动前检查** — 连库 + 查 `bookings.airports_data` 是否存在

```javascript
app.use(require('./routes/index'));
app.use(require('./routes/cities'));
app.use(require('./routes/airports'));
// ...
```

#### `db.js` — 连接池

```javascript
const pool = new Pool({
  host: process.env.DB_HOST || 'db',
  database: process.env.DB_NAME || 'demo',
  // ...
});
```

Docker 通过 `docker-compose.local.yaml` 注入 `DB_HOST=host.docker.internal`，连本机 Postgres。

#### `asyncHandler.js` — 防止 async 路由崩溃

```javascript
function asyncHandler(fn) {
  return (req, res, next) => {
    Promise.resolve(fn(req, res, next)).catch(next);
  };
}
```

所有 `async` 路由包一层，错误交给 `main.js` 的统一 handler，**单次查询失败不会拖垮整个进程**。

#### Docker

| 文件 | 用途 |
|------|------|
| `docker-compose.local.yaml` | **推荐**：只起 web 容器，数据库用本机 demo |
| `docker-compose.yaml` | 全栈：web + postgres（需自行导入 demo） |
| `app/Dockerfile` | Node 20 镜像，`/app` 为工作目录 |

改代码后：`docker compose restart web`（volume 挂载 `./app`，无需 rebuild）。

---

### 3.3 路由层 — 与 D5 一一对应

#### `routes/index.js` → `GET /`

健康检查，返回纯文本确认服务存活。

#### `routes/cities.js` → Location 前两接口

| 路由 | SQL 要点 |
|------|----------|
| `GET /cities` | `SELECT DISTINCT city->>'en' FROM bookings.airports_data` |
| `GET /cities/:city/airports` | 按 `city->>'en' = $1` 过滤，返回 `airport_code` + `airport_name` |

**新版 demo 关键：** 城市/机场名存在 **jsonb**，必须用 `->>'en'` 取英文。

#### `routes/airports.js` → Location 后三个接口（含筛选）

| 路由 | 函数 |
|------|------|
| `GET /airports` | 全表机场列表 |
| `GET /airports/:code/inbound` | `buildScheduleQuery({ side: 'inbound', placeFilter: origin, ... })` |
| `GET /airports/:code/outbound` | `buildScheduleQuery({ side: 'outbound', placeFilter: destination, ... })` |

**`buildScheduleQuery` 实现 D5 契约中的筛选逻辑：**

1. **地点过滤** — `isAirportCode()` 判断三字码：码则比 `r.arrival_airport` / `r.departure_airport`；否则比 `city->>'en'`
2. **时间过滤** — `time_from`/`time_to` 用正则校验 `HH:MM:SS`，拼 `r.scheduled_time >=/<= $n::time`
3. **条数限制** — `parseLimit()`：默认 10，上限 100
4. **时效航线** — `(SELECT MAX(actual_departure) FROM flights) <@ r.validity`，只取当前有效班表
5. **去重 + 排序** — 子查询 `DISTINCT ON (r.route_no)`，外层按 `scheduled_time` 排序后 `LIMIT`

**讲词：** 这里不是简单 `SELECT * FROM routes`，而是把 D5 里四个 query 参数翻译成动态 WHERE，并在 SQL 层控制返回量。

#### `routes/routes.js` → `GET /routes`（最复杂）

**功能：** 两点间航班搜索，支持直飞 + 多次中转。

**核心技术：** PostgreSQL **`WITH RECURSIVE`** 递归 CTE。

```
锚点（第一段）          递归（接下一程）              外层过滤
从 from 出发的航班  →  落地机场找下一班  →  ...  →  arrival = to
```

| 规则 | 常量/逻辑 |
|------|-----------|
| 转机最短间隔 | 40 分钟 |
| 转机最长等待 | 24 小时 |
| 防绕圈 | `path_airports` 数组记录途经机场 |
| 起点/终点 | `isAirportCode()` 区分三字码与城市名 |
| 舱位 | `bookingClass` → 检查机型是否有对应 `seats` |
| 时效 | `f.scheduled_departure <@ r.validity` |

**开发踩坑（已修复）：** 外层 WHERE 不能写 `i.arrival_city`（无别名 `i`），应直接用 `arrival_city`；JOIN 顺序需保证 `arr_a2` 在使用前已 JOIN。

**演示 URL：**

```
http://localhost:3000/routes?from=Moscow&to=LED&date=2025-12-31&connections=1&bookingClass=economy
```

#### `routes/bookings.js` → `POST /bookings`

**流程（单事务 `BEGIN` → `COMMIT` / `ROLLBACK`）：**

1. 校验 `segments` 非空
2. 每段 `resolveFlightId()` — 按 `routeNo` + 起降机场 + 精确出发时间匹配 `flights`
3. 算价 — `SUM(飞行分钟数 × RATE[fare_condition])`，RATE 与 D4 一致：`{ Economy: 50, Comfort: 65, Business: 100 }`
4. 生成唯一 `book_ref`（6 位）和 `ticket_no`（13 位，前缀 `000543`）
5. 写入 `bookings` → `tickets` → `segments` 三表

**讲词：** 三表必须同事务，任一失败全回滚；单价来自 D4 不是拍脑袋。

#### `routes/checkin.js` → `POST /check-in`

**流程：**

1. 查 `boarding_passes` — 已有则 **200** 返回（幂等）
2. 从 `segments` 取该票所有航段
3. 每段：查机型 → 在 `seats` 里找第一个未被占用的座位（`NOT EXISTS`）→ 写 `boarding_passes`
4. 登机时间 = 起飞前 30 分钟；登机序号 = 该航班当前 max + 1
5. 首次成功 **201**

### 3.4 请求链路一览

```
HTTP 请求
  → main.js（express.json）
  → routes/*.js（asyncHandler 包装）
  → pool.query（pg 连接池）
  → PostgreSQL bookings schema
  → res.json(...) 或 res.status(201).json(...)
  → 出错 → next(err) → main.js 统一 JSON 错误
```

### 3.5 D6 小结（答辩收束）

> D6 是 D5 契约的忠实实现：**6 个路由文件对应 8 个接口**；复杂逻辑集中在 `routes.js`（递归 CTE）和 `airports.js`（动态筛选 SQL）；订票/值机用事务保证一致性。  
> 架构刻意保持扁平，便于答辩时「Swagger 点接口 → 打开同名 JS → 指 SQL」。

## 六、Q&A

### Q1：D5 和 D6 各有一个 `api.yaml`，区别？

> D5 是设计稿；D6 与之同步，作为实现依据。改契约时两处一起改，并更新对应 `routes/*.js`。

### Q2：筛选参数为什么先在 D5 定义？

> 契约先行：参数名、类型、默认值、上限在 OpenAPI 里写死，前端看 Swagger 即可对接，后端有明确验收标准。

### Q3：`time_from`/`time_to` 格式错了怎么办？

> D5 定义了 pattern；D6 `airports.js` 校验失败返回 400 `{ "error": "time_from must be HH:MM:SS" }`。

### Q4：为什么不用 MVC？

> 8 个接口规模下，扁平 `routes/` 更直观；每个文件从路由到 SQL 一条线，答辩指读效率高。

### Q5：递归 CTE 干什么用？

> `/routes` 中转搜索：先找第一程，再在落地机场递归找下一程，直到到达终点或超过 `connections` 上限。

### Q6：为什么订票用事务？

> `bookings`、`tickets`、`segments` 三表必须同时成功或同时回滚。

### Q7：值机为什么只传 `ticket_no`？

> 票号足以反查航段；重复请求返回已有登机牌（幂等）。

### Q8：`validity @>` 是什么？

> 航线有时效版本；航班起飞时间必须落在 `validity` 区间内才能 join 到正确航线。

### Q9：50/65/100 的依据？（定价必问）

> 不是拍脑袋。demo 历史票价满足 **票价 = 每分钟单价 × 飞行分钟数**。对全部 `Arrived` 航段反推：

| 舱位 | 反推均价 |
|------|----------|
| Economy | **50.00** / 分钟 |
| Comfort | **65.00** / 分钟 |
| Business | **100.00** / 分钟 |

回代误差为 0。当场验证：`Task D4/verify_rates.sql`。
