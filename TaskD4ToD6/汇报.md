# 航空客运预订系统后端开发项目汇报

---

## 一、项目背景与目标

本项目基于 PostgresPro 提供的航空运输演示数据库（2025-09-01 版本），完成了三个递进式开发任务：

- **Task D4**：数据工程——从历史数据中还原定价规则
- **Task D5**：API 设计——基于 OpenAPI 3.0 规范设计 RESTful 接口
- **Task D6**：API 实现——使用 Node.js 实现完整的后端服务

项目的核心挑战在于：所使用的数据库版本（2025-09-01）相比旧版发生了多处 Schema 变更，需要在开发过程中严格遵守新版规范，避免使用已废弃的表名和字段。

---

## 二、数据库版本适配（重要）

本项目使用的是 2025-09-01 最新版 demo 数据库，与网络上大多数参考资料所用的旧版存在以下关键差异，这也是本次开发中最需要注意的地方：

| 变更项 | 旧版 | 新版（本项目） |
|--------|------|------|
| 航段表 | `ticket_flights` | `segments` |
| 价格字段 | `amount` | `price` |
| 飞机表 | `aircrafts` | `airplanes_data`（jsonb） |
| 机场表 | `airports`（视图） | `airports_data`（jsonb，无视图） |
| 飞机代码字段 | `aircraft_code` | `airplane_code` |
| 航线关联方式 | `routes.flight_no` | Temporal join：`routes.validity @> flights.scheduled_departure` |
| 飞机代码位置 | `flights.aircraft_code` | `routes.airplane_code`（需 join 获取） |
| 多语言支持 | 无 | `airport_name`、`city` 为 jsonb，需 `->>'en'` 提取 |

这些变更意味着所有 SQL 查询都必须重新设计，不能直接沿用旧版写法。

## 三、Task D4：数据修复与定价规则构建

### 任务目标

历史航班的部分座位缺少价格记录，需要从已有的 `segments` 历史数据中还原每条航线、每个舱位的标准定价，并建立 `pricing_rules` 表供未来航班使用。

### 实现思路

整个脚本分为以下几个步骤：

1. **提取历史均价**：从 `segments` 表 join `boarding_passes` 和 `flights`，计算每个 `(route_no, fare_conditions, seat_no)` 组合的历史平均价格。

2. **补全缺失价格**：对于没有历史数据（price = 0）的座位，用同一航线、同一舱位、同一排其他座位的价格进行插值补全。

3. **聚合定价规则**：按 `(route_no, fare_conditions)` 聚合，计算该组合的平均价格和座位数量，写入 `pricing_rules` 表。

4. **清理中间表**：脚本执行完毕后自动删除所有中间临时表，保持数据库整洁。

### 产出

```sql
bookings.pricing_rules (
  route_no        text,          -- 航线编号
  fare_conditions text,          -- 舱位等级
  base_price      numeric(10,2), -- 基准价格
  seat_count      bigint,        -- 参与统计的座位数
  PRIMARY KEY (route_no, fare_conditions)
)
```

同时为 `route_no` 字段建立了 B-Tree 索引，保障后续查询性能。

## 四、Task D5：RESTful API 设计

### 设计原则

- 遵循 RESTful 风格，资源名称使用复数名词
- 使用标准 HTTP 状态码（200、201、400、404、500）
- 统一错误响应格式：`{ "message": "错误描述" }`
- 规范文件采用 OpenAPI 3.0 格式，可直接导入 Swagger UI 预览

### 接口设计

共设计 7 个接口，覆盖城市、机场、航班、预订、值机五个业务域：

| 接口 | 说明 | 关键设计决策 |
|------|------|------|
| `GET /cities` | 分页获取城市列表 | 加 DISTINCT 去重，按字母排序 |
| `GET /cities/{cityName}/airports` | 城市下的机场 | 城市名不存在返回 404 |
| `GET /airports` | 分页获取机场列表 | 返回完整机场信息含坐标 |
| `GET /airports/{code}/schedules` | 进港/出港时刻表 | `type` 参数区分方向，使用 temporal join |
| `GET /flights` | 航班搜索 | 支持城市名或机场代码，支持中转次数限制 |
| `POST /bookings` | 创建预订 | 事务操作，自动生成 bookRef 和 ticketNo |
| `POST /flights/checkins` | 办理值机 | 自动分配座位，生成登机牌 |

一个值得说明的设计决策：原始设计中 `/flights` 使用 `GET + requestBody`，这在 OpenAPI 3.0 中是不推荐的做法。本项目将搜索参数全部改为 query string，符合 HTTP 语义规范。

## 五、Task D6：API 实现

### 架构设计

采用经典的三层架构：

```
路由层 (routes/)
    ↓ 入参校验 (middlewares/validators/)
控制器层 (controllers/)
    ↓
模型层 (models/)  ←→  PostgreSQL
```

- **路由层**：仅负责 URL 映射和中间件挂载
- **控制器层**：薄层，只做参数提取和响应格式化
- **模型层**：承载所有业务逻辑和 SQL 查询
- **中间件**：统一处理校验、日志、错误

### 核心实现细节

#### 1. 参数化查询（SQL 注入防护）

所有 SQL 查询均使用 `$1, $2...` 占位符，杜绝 SQL 注入风险：

```javascript
// 错误做法（原始代码存在此问题）
`WHERE city = '${cityName}'`

// 正确做法（本项目实现）
`WHERE city->>'en' = $1`, [cityName]
```

#### 2. Temporal Join（时序关联）

这是新版 Schema 最重要的变化。`routes` 表的路由会随时间变化，必须用时间范围匹配：

```sql
JOIN bookings.routes AS r
  ON r.route_no = f.route_no
  AND r.validity @> f.scheduled_departure
```

如果不加 `r.validity @> f.scheduled_departure` 条件，同一航线的多个历史版本会导致结果重复或错误。

#### 3. 预订事务（ACID 保证）

创建预订涉及多表写入，使用数据库事务保证原子性：

```
BEGIN
  → 查询航班状态（检查是否已到达/取消）
  → 计算剩余座位数（检查是否已满）
  → 获取票价（优先从 pricing_rules，回退到 segments 历史数据）
  → INSERT bookings
  → INSERT tickets
  → INSERT segments
COMMIT / ROLLBACK（任一步骤失败则全部回滚）
```

#### 4. 航班搜索（递归 CTE）

中转航班搜索使用 PostgreSQL 递归公共表表达式（WITH RECURSIVE），支持最多 3 次中转：

- 基础查询：从出发机场出发的所有直飞航班
- 递归步骤：从上一段的到达机场继续搜索，要求转机间隔 ≥ 30 分钟
- 终止条件：到达目的机场，或超过最大中转次数
- 防环处理：通过 `path_ids` 和 `path_airports` 数组避免重复经过同一航班/机场

#### 5. 定价回退机制

`pricing_rules` 表需要先执行 Task D4 脚本才存在。为了让 API 在没有该表时也能正常工作，使用 PostgreSQL SAVEPOINT 实现优雅回退：

```javascript
try {
  await transaction.query('SAVEPOINT price_lookup');
  // 尝试从 pricing_rules 查询
  await transaction.query('RELEASE SAVEPOINT price_lookup');
} catch (_) {
  // 表不存在，回滚到 savepoint，从 segments 历史数据取价
  await transaction.query('ROLLBACK TO SAVEPOINT price_lookup');
}
```

## 六、遇到的主要问题与解决方案

### 问题 1：数据库 Schema 与旧版完全不同

**现象**：项目原始代码使用旧版表名（`ticket_flights`、`aircrafts`、`airports` 视图），在新版数据库上全部报错。

**解决**：逐一排查所有 SQL 查询，按新版 Schema 重写，特别是：
- 用 `airports_data` 替代 `airports` 视图，并用 `->>'en'` 提取 jsonb 字段
- 用 `segments` 替代 `ticket_flights`，`price` 替代 `amount`
- `airplane_code` 从 `flights` 表移到了 `routes` 表，需要 join 获取

### 问题 2：Temporal Join 导致结果重复

**现象**：不加时间范围条件时，同一航线的多个历史路由版本会产生重复结果。

**解决**：所有涉及 `routes` 表的查询都加上 `r.validity @> f.scheduled_departure` 条件。

### 问题 3：时区导致日期匹配失败

**现象**：服务器时区为 UTC+8，数据库存储的是 UTC 时间。用 `::date` 截断时，`2025-11-30 17:00 UTC` 会被截断为 `2025-12-01`（本地时间），导致按日期查询时结果不一致。

**解决**：统一使用 PostgreSQL 的 `::date` 转换（按数据库服务器本地时区），测试用例中的日期也相应调整为本地时区视角的日期（`2025-12-01`）。

### 问题 4：SQL 注入风险

**现象**：原始代码中大量使用字符串拼接构造 SQL，存在安全隐患。

**解决**：全部改为参数化查询（`$1, $2...` 占位符），彻底消除 SQL 注入风险。

### 问题 5：pricing_rules 表不存在时预订失败

**现象**：Task D4 脚本未执行时，`pricing_rules` 表不存在，预订接口直接报 500 错误。

**解决**：使用 PostgreSQL SAVEPOINT 机制，查询失败时自动回退到从 `segments` 历史数据取价，保证接口在任何情况下都能正常工作。

## 七、性能优化

为保障复杂查询（尤其是航班搜索和值机）的响应速度，建议在数据库中创建以下索引：

```sql
-- Temporal join 核心索引
CREATE INDEX idx_flights_route_departure
  ON bookings.flights (route_no, scheduled_departure);

-- 航段查询
CREATE INDEX idx_segments_flight_fare
  ON bookings.segments (flight_id, fare_conditions);

-- 登机牌查询
CREATE INDEX idx_boarding_passes_flight ON bookings.boarding_passes (flight_id);
CREATE INDEX idx_boarding_passes_ticket ON bookings.boarding_passes (ticket_no);

-- 座位查询
CREATE INDEX idx_seats_airplane_fare ON bookings.seats (airplane_code, fare_conditions);

-- jsonb 城市名查询（表达式索引）
CREATE INDEX idx_airports_city_en ON bookings.airports_data ((city->>'en'));
```

本项目完整实现了航空客运预订系统的后端服务，主要成果：

1. **Task D4**：成功从历史数据还原定价规则，建立了 `pricing_rules` 表，覆盖所有航线和舱位组合。

2. **Task D5**：设计了符合 OpenAPI 3.0 规范的 7 个 RESTful 接口，涵盖城市、机场、航班搜索、预订和值机全流程。

3. **Task D6**：实现了完整的 Node.js 后端服务，19 个集成测试全部通过，接口功能正常，代码安全（参数化查询）、健壮（事务、错误处理、回退机制）。

项目最大的技术难点是适配 2025-09-01 新版数据库 Schema，特别是 Temporal Join 的正确使用和 jsonb 字段的处理，这两点在旧版参考资料中均无法找到，需要通过阅读官方文档和实际调试解决。
