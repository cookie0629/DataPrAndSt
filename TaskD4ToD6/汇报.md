# 航空客运预订系统 

---

## 一、答辩

### 【开场白】

> 做的是一个**航班查询与预订的后端引擎**：用户能查城市、机场、航线，能订票，能在线值机。  
> 项目分三步走：**D4 在数据库里补定价规则，D5 用 OpenAPI 定接口契约，D6 用 Node.js 实现并做了架构精简**。

---

### 【Task D4】数据库底层

#### 1. 先讲问题

> D4 要解决的问题是：demo 库里只有历史售票记录（`segments` 表里的 `price`），**没有一张统一的定价规则表**，D6 订票时就没法按统一规则算价。  
> 所以我要新建 `bookings.pricing_rules`，从历史数据里统计出每条航线、每种舱位的定价参数。

#### 2. 指文件结构

打开 **`Task D4/`**，说明两个文件分工：

| 文件 | 作用 |
|------|------|
| `price.sql` | 建表 + 灌数据（真正干活） |
| `d4.sql` | 跑查询验证定价是否合理 |

#### 3. 现场演示

**（终端执行，或提前准备好截图）**

```powershell
psql -U postgres -d demo -f "Task D4/price.sql"
psql -U postgres -d demo -f "Task D4/d4.sql"
```

讲词：

> 先执行 `price.sql` 创建并填充定价表；再执行 `d4.sql`，会列出「按规则算出来的价格」和「数据库里实际票价」的对比，用来验证模型是否合理。

#### 4. 指代码讲表关联— 打开 `Task D4/price.sql`

**INSERT … SELECT** 部分：

> 这里把三张表串起来：  
> - `routes`：航线定义，有起降机场、飞行时长；  
> - `flights`：具体某天的航班实例；  
> - `segments`：这张票在某个航班上的售价。  
>
> **重点在这一行 JOIN 条件**：
>
> ```sql
> AND r.validity @> f.scheduled_departure
> ```
>
> 新版库里，同一条 `route_no` 在不同时间段可能是不同版本（不同机型、不同配置），`validity` 是一个时间段。  
> 航班起飞时间必须落在这个时间段里，才能 join 到**正确那一版**航线。如果只写 `route_no` 相等，会匹配到多行历史版本，结果就错了。

**CASE 定价**：

> 经济舱每分钟 50、舒适舱 65、商务舱 100——这个规则和 D6 订票代码里的 `RATE` 常量是一致的，D4 定规则、D6 用规则。

#### 5. 指索引— 打开 `Task D6/postgres-init/indexes.sql`

> 索引不是 D4 建的，但和查询性能直接相关。  
> 比如 `idx_flights_departure_time` 对应航线搜索里按日期筛航班；`idx_boarding_passes_ticket` 对应值机按票号查登机牌。  
> 可以理解为**给常查的列加目录**，避免每次全表扫描。

#### 第三部分：演示 ② 航线搜索 + 指递归
**浏览器操作：**

```
http://localhost:3000/routes?from=Moscow&to=LED&date=2025-12-31&connections=1&bookingClass=economy
```

> 这是项目最难的接口。用户给起点、终点、日期，可以限制最多转几次机。  
> 返回的每条结果有 `connectionsCount`（中转次数）和 `segments`（每一段的航班信息）。

**打开 `app/routes/routes.js`，分三块指：**

**（1）参数处理（文件上部）**

```javascript
const maxConn = connections === 'unbound' ? 10 : parseInt(connections ?? '0', 10);
// isAirportCode：三字码走机场字段，否则走城市名字段
```

> `from`/`to` 既可以是 `SVO` 这种三字码，也可以是 `Moscow` 这种城市名，代码里用正则判断后拼不同的 WHERE 条件。

**（2）递归 CTE 锚点（`WITH RECURSIVE itins AS` 第一段 SELECT）**

> 第一段是**锚点**：找「当天从起点出发」的所有第一程航班，`connections = 0`。

**（3）递归部分（`UNION ALL` 后面）**

> 第二段是**递归**：在上一程落地的机场，找能接上的下一程。  
> 约束有三条我要能说出来：  
> - 转机至少间隔 **40 分钟**；  
> - 最多等 **24 小时**；  
> - `path_airports` 数组记录已飞过的机场，**防止绕圈**。

**（4）第二次查询拼 segments**

> 递归结果只有 `flight_id` 数组，后面再查一次 flights 表，拼成 api.yaml 里定义的 JSON 格式返回。

#### 第四部分：演示 ③ 订票 + 指事务（1 分钟）

**（可用 Postman / curl / Swagger 发 POST）**

```http
POST http://localhost:3000/bookings
Content-Type: application/json

{
  "passenger_id": "1234 567890",
  "passenger_name": "IVAN IVANOV",
  "fare_condition": "Economy",
  "outbound": true,
  "segments": [
    {
      "routeNo": "PG0370",
      "departureAirportCode": "OVB",
      "arrivalAirportCode": "SVO",
      "departureTime": "2025-12-30 15:35:00+00"
    }
  ]
}
```

成功应返回 `book_ref` 和 `ticket_no`（201）。

**指 `app/routes/bookings.js`：**

```javascript
await client.query('BEGIN');
// 1. resolveFlightId — 把前端 segments 转成 flight_id
// 2. 算总价：飞行分钟数 × RATE[舱位]
// 3. INSERT bookings → tickets → segments
await client.query('COMMIT');
// 失败则 ROLLBACK
```

讲词：

> 订票涉及三张表，必须用**事务**包起来：要么全部写入成功，要么全部回滚，不会出现「有订单没票」的脏数据。  
> 票价公式和 D4 一致：`时长(分钟) × 单价`，Economy 是 50。

---

#### 第五部分：演示 ④ 值机

**POST `http://localhost:3000/check-in`**

```json
{ "ticket_no": "上一步返回的票号" }
```

**指 `app/routes/checkin.js` 三步：**

1. 先查 `boarding_passes` — 已值机则 **200** 直接返回（幂等）  
2. 从 `segments` 拿这张票有哪些航班  
3. 在 `seats` 里找该舱位第一个空座 → 写入 `boarding_passes` → **201**

> 登机时间设为起飞前 30 分钟；`boarding_no` 是该航班当前最大序号 +1。

## 四、刁钻问题 Q&A

### Q1：为什么 join 要写 `validity @> scheduled_departure`？

**考察意图**：是否理解时效航线，不是死记 SQL。

**大白话**：

> 同一条航线号在不同时间段可能是不同版本。`validity` 标记这条记录什么时候有效，航班起飞时间必须落在这个区间里，join 到的才是「当时真正飞的那条航线」。只用 `route_no` 相等会把历史版本全 join 进来。

---

### Q2：递归 CTE 是什么？为什么不用程序里写循环？

**考察意图**：最难 SQL 是否真懂。

**大白话**：

> 中转搜索是「图上的路径搜索」：先找第一程，再在落地机场找下一程，一层层扩展。  
> PostgreSQL 的 `WITH RECURSIVE` 专门干这个，比在 Node 里多层循环查库更清晰，也减少往返数据库次数。  
> `connections=0` 时锚点查完就结束，相当于只搜直飞。

---

### Q3：订票为什么要事务？

**考察意图**：数据一致性。

**大白话**：

> 要写 `bookings`、`tickets`、`segments` 三张表。中间任何一步失败，事务 ROLLBACK，不会留下半拉子订单。这是订票系统的底线。

---

### Q4：值机为什么只传 `ticket_no`？

**考察意图**：接口设计与表关系。

**大白话**：

> 票号在业务上已经能定位乘客和行程，`segments` 表有 `(ticket_no, flight_id)`。不让用户传内部 `flight_id` 是更好的 API 设计。座位自动分配第一个空位，接口简单；重复值机返回 200 已有登机牌，不会占两次座。

---

### Q5：为什么拆掉 MVC？不算不规范吗？

**考察意图**：架构取舍（加分项）。

**大白话**：

> MVC 适合大团队长期维护。我这个作业 8 个接口，多层跳转反而增加理解成本。参考课程模板后改成 `main.js + routes/`，一个文件一组 API，SQL 和逻辑在一起，答辩和排查都更直接。规模大了再拆层完全来得及。

---

### Q6：`airports_data does not exist` 是什么原因？

**考察意图**：环境是否跑通（结合你的真实踩坑）。

**大白话**：

> Docker 自带的 Postgres 只是空库 `demo`，没有 PostgresPro 的航班表。必须先导入 demo 数据，或用 `docker-compose.local.yaml` 连本机已导入 demo 的 PostgreSQL。Docker 开着不等于数据已经导入。

---

### Q7：索引怎么选的？

**考察意图**：索引原理。

**大白话**：

> 看 SQL 的 WHERE 条件：`scheduled_departure` 筛日期、`departure_airport`/`arrival_airport` 筛机场、`ticket_no` 查登机牌——这些列建索引。不能乱加，写入和维护都有成本。
